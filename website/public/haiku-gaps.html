Gap 1: Cross-Intent Triggers

Business processes chain across studios. Today, each intent is an island.

Scenario: New Customer Lifecycle

A sales deal closes. Customer success should automatically begin onboarding. But today, someone has to manually create the CS intent.

Sales: close → ??? trigger ??? → CS: onboarding → CS: adoption
Scenario: Product to Engineering

Product strategy prioritizes a feature. Software studio should receive the roadmap item as an intent seed. No automated handoff exists.

Product: roadmap → ??? trigger ??? → Software: inception → Software: design
Scenario: Incident to Prevention

Incident postmortem produces action items. These should automatically create compliance or software intents for prevention work.

Incident: postmortem → ??? trigger ??? → Software: inception
Incident: postmortem → ??? trigger ??? → Security: reconnaissance

Proposed Fix Studio-Level Triggers

Add a triggers: section to STUDIO.md that declares what happens when an intent completes.

# sales/STUDIO.md
triggers:
  on-complete:
    - studio: customer-success
      template: new-customer-onboarding
      map:
        company: "{{ intent.title }}"
        deal-context: "{{ stages.close.outputs.handoff }}"
      condition: "{{ stages.close.outcome == 'won' }}"

Triggers fire after the final stage gate passes. The map: section pipes outputs from the source intent into the target template's parameters. The condition: field prevents triggers when the intent doesn't meet criteria (e.g., lost deals don't trigger onboarding).

Gap 2: Intent Templates / Playbooks

Repeatable processes shouldn't require fresh elaboration every time.

Today Studios

Define the lifecycle: which stages, which hats, which gates.

  • Studio = lifecycle shape
  • Stage = phase definition
  • Each intent starts from scratch

Missing Intent Templates

Pre-filled intents with standard units, criteria, and references. Instantiate with parameters.

  • "New Customer Onboarding for {company}"
  • "Quarterly Compliance Evidence Collection"
  • "Product Launch Campaign for {product}"
  • "Incident Response for {service}"

Proposed Fix Studio Templates Directory

Add templates/ to the studio directory. Each template is a pre-filled intent with parameterized fields.

# customer-success/templates/new-customer-onboarding.md
---
name: new-customer-onboarding
studio: customer-success
parameters:
  - name: company
    required: true
  - name: deal-context
    required: false
units:
  - name: "technical-setup"
    stage: onboarding
    criteria:
      - "{{ company }} environment provisioned and accessible"
      - "SSO/auth configured and tested"
  - name: "kickoff-meeting"
    stage: onboarding
    criteria:
      - "Kickoff scheduled with {{ company }} stakeholders"
      - "Success milestones agreed and documented"
---

Templates are instantiated via /haiku:new --template new-customer-onboarding --company "Acme Corp". Parameters fill in the template. Triggers can also instantiate templates automatically.

Proposed Fix Cross-Studio Inputs

Extend the stage inputs: schema to reference outputs from other studios via an intent: field.

# software/stages/inception/STAGE.md
inputs:
  - intent: "{{ triggers.source }}"
    studio: product-strategy
    stage: roadmap
    output: roadmap-item
  - intent: "{{ triggers.source }}"
    studio: product-strategy
    stage: user-research
    output: research-insights

When an intent is triggered by another studio, the source intent's slug is available via triggers.source. The stage can then pull outputs from that specific intent. For manually created intents, the user specifies which source intent to bind to.

This also enables a knowledge graph — the orchestrator can trace the full chain of intents and their data dependencies across studios.

Gap 4: People Layer

H·AI·K·U is single-agent. Business operations involve teams, handoffs, and approval chains.

Missing Role Assignment

Stages need owners — a person or team responsible for driving that stage's work and approving its gate.

  • Sales proposal: account executive
  • Software security: security team
  • Compliance certify: compliance officer

Missing Approval Chains

Gates are binary approve/reject. Business needs multi-level, conditional, and time-bound approvals.

  • Team lead → director → VP
  • "Approved, but change X first"
  • Auto-escalate if no response in 48h

Missing Handoff Protocols

When one team's studio completes, context must transfer to the next team cleanly.

  • Sales → Customer Success handoff
  • Product → Engineering handoff
  • Context, commitments, expectations

Missing Event-Driven Gates

The await gate is manual. Real business needs webhook-driven advancement.

  • Email reply received → advance negotiation
  • Contract signed → advance close
  • CI pipeline passed → advance deployment

Proposed Fix Stage Ownership + Gate Protocols

Add owner: to STAGE.md and extend the gate schema to support multi-level approval.

# software/stages/security/STAGE.md
owner: security-team
review: external
gate-protocol:
  approvers: [security-lead, ciso]
  mode: sequential          # sequential | any | all
  timeout: 48h
  timeout-action: escalate  # escalate | auto-advance | block
  conditions:
    - "no HIGH findings from review agents"

# sales/stages/proposal/STAGE.md
owner: account-executive
review: [ask, await]
gate-protocol:
  await-event: customer-response
  webhook: /api/events/customer-reply
  timeout: 7d
  timeout-action: escalate

owner: identifies who drives the stage. gate-protocol: extends the simple gate type with multi-level approval, event integration, timeouts, and escalation. The webhook: field enables external events to advance await gates automatically.

Gap 5: Portfolio Visibility

Leadership running 50 intents across 5 studios needs a single view.

Missing Portfolio Dashboard

  • All active intents by studio and stage
  • Bottleneck detection (work piling up at gates)
  • Health indicators per intent
  • Cross-studio dependency visualization

Missing Cross-Studio Metrics

  • Velocity trends across studios
  • Gate pass/fail rates over time
  • Stage duration benchmarks
  • Conversion rates (sales), quality scores (software)

Proposed Fix Portfolio API + Dashboard

A /haiku:portfolio skill that aggregates across all intents, plus a web dashboard endpoint.

# .haiku/portfolio.json (auto-maintained by orchestrator)
{
  "intents": {
    "acme-onboarding": {
      "studio": "customer-success",
      "active_stage": "adoption",
      "owner": "cs-team",
      "health": "on-track",
      "triggered_by": "acme-deal",
      "stages_complete": 2,
      "stages_total": 5
    },
    "q2-compliance": {
      "studio": "compliance",
      "active_stage": "remediate",
      "owner": "compliance-officer",
      "health": "at-risk",
      "blocked_at": "remediate (await: vendor response)",
      "stages_complete": 2,
      "stages_total": 5
    }
  },
  "metrics": {
    "active_intents": 12,
    "blocked_intents": 3,
    "avg_stage_duration_days": 4.2,
    "gate_pass_rate": 0.87
  }
}

The orchestrator updates portfolio.json on every stage transition. The web dashboard reads this file. /haiku:portfolio renders a CLI summary. For multi-repo organizations, a central aggregator collects portfolio files across projects.

Summary

Built (Single-Project Foundation)

  • 12 studios, 60 stages, 120+ hats
  • 72 review agents with cross-stage includes
  • 4 review gate types (auto/ask/external/await)
  • 38 operation templates, 26 reflection dimensions
  • Stage-scoped refinement for upstream gaps
  • Continuous + discrete execution modes
  • One PR per intent delivery model
  • Quality gates via backpressure

Missing (Multi-Company Platform)

  • Cross-intent triggers (intent chains)
  • Intent templates / playbooks
  • Cross-studio data flow
  • Event-driven gate advancement
  • Role assignment and ownership
  • Multi-level approval workflows
  • Cross-team handoff protocols
  • Portfolio dashboard and metrics
  • Priority / resource management
  • Immutable audit trail

The lifecycle layer is solid. The orchestration, people, and visibility layers are the gap between a project tool and a business platform.
