Design Spec · unit-06

Revisit confirmation modal

Destructive-action confirmation before haiku_revisit fires. Approve stays single-click (reversible by next gate); Request Changes always routes through this modal.

Overview Anatomy Variant A · Gate-invoked Variant B · User-opened Variant C · Typed feedback Light / Dark Mobile narrow Focus / A11y Copy Tokens

Overview

Trigger

Reviewer clicks Request Changes, types free-form feedback, or opens the review app mid-session.

Why a modal

haiku_revisit rewrites stage state, re-queues downstream units, and pushes to the main intent branch. Not reversible from the UI.

Approve contrast

Approve is single-click. It is reversible by the next review gate. Revisit is not — hence the modal.

Anatomy

Information hierarchy top-to-bottom: target → downstream impact → new feedback preview → open feedback in scope → footer actions. Numbered labels map to the spec IA in the unit definition.

Mobile adaptations

  • Viewport < 640px: modal fills screen with 8px insets. inset-x-2 top-2 bottom-2.
  • Footer stacks vertically; Confirm is primary and first (top), Cancel secondary below. On iPhone SE the primary stays within the bottom 1/3 reachable zone.
  • Every tappable element ≥ 44×44px. The header ✕ close button is explicit on mobile (on desktop it's implicit via backdrop-click + ESC).
  • Confirm label wraps into two lines at narrow widths: stage name on line 1, side-effect summary on line 2, text-xs font-medium opacity-90.
  • Body region scrolls independently; footer is sticky at the bottom of the modal — reviewer always sees the action.

Focus trap, return focus, dismiss paths

Focus order (tab-cycle)

  1. On open: focus moves to Cancel (safe default — no accidental Enter-to-confirm for destructive action).
  2. Tab → Confirm & Revisit button.
  3. Tab → cycles back to Cancel (focus trap).
  4. Shift+Tab reverses the cycle. Focus never escapes the modal.
  5. On close: focus returns to the element that triggered the modal — the Request Changes button in the sidebar (or the keyboard shortcut originator).

Dismiss paths (all non-destructive)

  • Esc Closes modal. No state change. Returns focus to trigger.
  • click backdrop Closes modal. No state change. Returns focus to trigger.
  • [Cancel] Closes modal. No state change. Returns focus to trigger.
  • [✕] (mobile) Explicit header close on viewports < 640px. Same effect as Cancel.

ARIA semantics

<div
  role="dialog"
  aria-modal="true"
  aria-labelledby="revisit-modal-title"
  aria-describedby="revisit-modal-lede"
>
  <h2 id="revisit-modal-title">Confirm revisit</h2>
  <p id="revisit-modal-lede">This will invoke haiku_revisit, rewriting state…</p>
  ...
</div>
  • • Body scroll is locked while the modal is open (body.overflow = 'hidden'); restored on close.
  • • The destructive Confirm button is not autofocus — focus lands on Cancel to prevent keyboard-confirm-by-accident.
  • • Screen reader announces title + lede when the dialog opens. Target stage + downstream count are part of the confirm button accessible name so AT users hear the full side-effect summary.
  • • Backdrop is aria-hidden="true"; it's decorative — the dialog owns the focus.

Approve contrast (documented)

Approve is a single-click primary action in the sidebar. It uses bg-teal-600 text-white and fires haiku_approve directly — no modal. The next review gate can still reject, so the action is reversible. Revisit cannot be undone from the UI, so it routes through this modal.
