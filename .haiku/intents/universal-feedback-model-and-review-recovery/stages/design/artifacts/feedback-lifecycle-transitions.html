Feedback Lifecycle — Ownership Matrix & Transition Spec

Unit 05 · FB-02 · Design artifact · Visual state machine + who-owns-what rules

Agent-owned Human-owned

1. State Machine

Four statuses, seven transitions. Teal edges are agent-driven (FSM / quality gate). Blue edges are user-initiated from the review UI. Dashed edges reopen a terminal state.

Teal = agent-owned

These transitions are driven by the orchestrator or the quality gate. The UI never exposes them as buttons.

Blue = human-owned

These transitions fire from review-UI footer buttons. Dashed edges are reopens (a terminal state going back to pending).

2. Ownership Matrix (rows = from · columns = to)

Every cell is either AGENT, USER, or — (not a valid transition). User-initiable cells are the only ones that render as buttons in the review UI.

from \ to pending addressed rejected closed notes
pending — (self) AGENT T1 · fix lands USER T4 · Dismiss AGENT T3 · quality gate User can dismiss a pending item. Only the agent can mark it addressed or close it via gate verification. Not user-initiable: pending → addressed, pending → closed.
addressed USER T7 · Reopen — (self) — USER T2 · Verify & Close Reviewer verifies the fix or reopens if the fix is wrong. Not possible: addressed → rejected (dismiss only applies to pending).
rejected USER T5 · Reopen — — (self) — Terminal until reopened. Agent cannot un-reject — only a human reconsiders a dismissal.
closed USER T6 · Reopen — — — (self) Terminal until reopened. Both agent-closed (gate verify) and human-closed (Verify & Close) collapse into the same closed row.

3. UI Contract — which transitions surface as buttons

This is the definitive list the review-app must render. If a transition is not on this list, the UI MUST NOT expose a button for it — even if the backing tool call would technically accept it.

User-initiable (MUST render as footer buttons)

  • pending → rejected · Dismiss · secondary button
  • addressed → closed · Verify & Close · primary-green button
  • addressed → pending · Reopen · secondary button
  • rejected → pending · Reopen · secondary button
  • closed → pending · Reopen · secondary button

NEVER user-initiable (MUST NOT render)

  • pending → addressed · agent marks when the fix lands during execute
  • pending → closed · quality gate closes structurally (e.g. the feedback references a file that no longer exists)

These transitions are informational only. The card MAY show status advancement as it happens, but MUST NOT offer a button for the user to trigger them.

4. Tool-layer enforcement (handoff note)

Scope of this unit: the UX contract above — which transitions surface as buttons, what their labels are, and their footer layout. See feedback-card-states.html for the per-state button inventory.

Out of scope (delegated to product / development): the author-type guard on the haiku_feedback_update tool (and equivalents) that enforces this split server-side. The UX contract above is the client contract. The tool layer MUST also reject any agent-forbidden transition regardless of origin — a malicious client cannot bypass ownership by skipping the UI.

Guard outcomes that produce toasts: see the optimistic-UI toast copy in feedback-card-states.html §6.
