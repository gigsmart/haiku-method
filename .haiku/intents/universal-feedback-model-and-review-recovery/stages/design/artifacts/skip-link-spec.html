Skip-link spec

Canonical visually-hidden-until-focused skip link pattern for every sticky-header layout. Closes FB-30. Reference behavior: pressing Tab on any affected artifact makes the skip link visible in the top-left of the viewport.

1. The pattern

Every artifact with a sticky top-0 header gets two skip links as the first focusable elements inside <body>. They are visually hidden (via sr-only) until focused, then appear in the top-left corner. 

<body>
  <!-- FB-30: Skip links. First focusable elements; visually hidden until focused. -->
  <a href="#main-content"   class="sr-only focus:not-sr-only focus:fixed focus:top-2 focus:left-2  focus:z-[200] focus:px-3 focus:py-2 focus:bg-teal-600 focus:text-white focus:rounded focus:shadow-lg focus:outline-none focus:ring-2 focus:ring-white">Skip to main content</a>
  <a href="#feedback-list"  class="sr-only focus:not-sr-only focus:fixed focus:top-2 focus:left-44 focus:z-[200] focus:px-3 focus:py-2 focus:bg-teal-600 focus:text-white focus:rounded focus:shadow-lg focus:outline-none focus:ring-2 focus:ring-white">Skip to feedback list</a>

  <header class="sticky top-0 …">…</header>

  <main id="main-content" …>
    …
    <aside id="feedback-list" …>…</aside>
  </main>
</body>
  • The two links are both inside <body> as the first children, before <header>, so they hit the Tab order first.
  • Both anchors (#main-content, #feedback-list) MUST exist on every affected artifact. Missing anchors break the skip link silently.
  • The second link lives at focus:left-44 (≈ 11rem) so it doesn't overlap the first when both are focused sequentially; they are never shown together.
  • The link background (teal-600) with white text meets WCAG AA contrast in both light and dark modes (contrast ratio ≈ 4.83:1 against white, and the teal ring on focus offsets it against any header below).

6. Why a visible skip link, not just the shortcut map

The g+key sequence in the shortcut map jumps to tabs, not to main content or the feedback list. Even if it did, keyboard-native alternatives don't replace the WCAG 2.4.1 Bypass Blocks requirement for first-time users who haven't opened the help overlay.

  • First-time users who don't know shortcuts need a discoverable bypass.
  • Screen-reader users in browse mode may not trigger the app's shortcut handler at all (see keyboard-shortcut-map.html §3b). A native <a href> skip link works with the SR browse mode.
  • Users with switch devices or head-tracking rely on discrete focusable targets; they cannot send a two-key chord.

7. Accessibility notes

  • WCAG 2.4.1 Bypass Blocks — the skip links satisfy this success criterion for every sticky-header layout.
  • Focus indicator — the link itself is the focus indicator. No additional ring needed; the teal background with white outline-offset ring is a high-contrast self-identifying focus target.
  • No motion — activation jumps focus via the browser's native anchor-link behavior. No smooth-scroll override, so users with prefers-reduced-motion are unaffected.
  • Screen-reader announcement — the link text is read as-is. No aria-label override (the visible text already says "Skip to main content" / "Skip to feedback list").
