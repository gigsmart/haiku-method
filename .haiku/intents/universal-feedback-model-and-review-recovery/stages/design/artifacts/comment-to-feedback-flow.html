Inline Rendering of Pre-existing Feedback

When pre-existing feedback has a source_ref targeting a text paragraph, the annotation card renders inline adjacent to that paragraph, just like user-created annotations. The highlight uses the same code-line-annotated background.

This means feedback items cannot be individually tracked, addressed, or closed.

Behavioral Notes

  • Pre-existing items load on mount via GET /api/feedback/{intent}/{stage}
  • Items from prior visits render in the "Visit N" groups in the sidebar; current visit items appear under "Current Visit"
  • The visit counter badge (e.g., "3x") uses escalation tiers: 2-3 = stone, 4-5 = amber, 6+ = red
  • Read-only items show status change buttons (Close, Reject, Reopen) but NOT Edit or Delete
  • Multiple annotations on the same paragraph stack vertically below the highlighted text

6. Error States

Three error categories: network failure on CRUD, concurrent edit conflict, and session expiry.

6a. Network Failure on CRUD

POST, PATCH, or DELETE fails due to network timeout or server error.

Optimistic Rollback

1. User triggers action (e.g., click "Close")

2. Badge updates optimistically (pending → closed)

3. API returns error → badge reverts (closed → pending)

4. Error toast appears at bottom of sidebar

Focus Indicators & Keyboard Nav

  • All interactive cards: focus:ring-2 focus:ring-teal-500 focus:ring-offset-2 dark:focus:ring-offset-stone-900 outline-none
  • Tab: moves through sidebar focus order (segmented control → filter pills → feedback items → action buttons → input → decision buttons)
  • Enter / Space: expand/collapse card, trigger action buttons
  • Escape: collapse expanded card, close bottom sheet, dismiss confirm dialog
  • Cmd+Enter: submit comment from textarea
  • Feedback items use role="listitem" in role="list"
  • Sidebar tabs use role="tablist" with aria-selected
  • Status badges include aria-label="Status: pending"
  • Status changes trigger role="status" aria-live="polite" announcements
  • Mobile bottom sheet traps focus when open; Escape closes it
  • Mobile FAB: aria-label="Open feedback panel, N pending"

Status Lifecycle (Optimistic UI)

All status transitions use optimistic updates. The badge changes instantly, then rolls back if the API call fails.

  draft ──────────────────── [Submit] ──────────────────── pending
                                                              │
                                                              ├── [Close] ──── closed
                                                              │                  │
                                                              │                  └── [Reopen] ── pending
                                                              │
                                                              ├── [Reject] ─── rejected
                                                              │                  │
                                                              │                  └── [Reopen] ── pending
                                                              │
                                                              └── [Agent addresses] ── addressed
                                                                                        │
                                                                                        ├── [Close] ── closed
                                                                                        └── [Reopen] ── pending

Optimistic Update Pattern

  1. User clicks action button (e.g., "Close")
  2. UI immediately updates badge to new status + triggers feedback-status-change animation
  3. PATCH /api/feedback/{intent}/{stage}/{id} with { status: "closed" }
  4. On success: no further UI change needed
  5. On error: revert badge to previous status + show error toast + trigger error-shake animation
  6. Live region announces: "Feedback FB-03 marked as closed" (or "Failed to update feedback FB-03")
