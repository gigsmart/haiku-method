Focus Ring Specification

Closes FB-37. Canonical rule for every interactive element across every artifact.

Tab through the demos below to see each ring render in place.

§1 — Canonical ring (applies by default)

2px solid ring, 2px offset, color teal-500 (#14B8A6) in light mode, teal-400 (#2DD4BF) in dark mode. Triggered by :focus-visible only — no ring on mouse click.

Applies to: every <button>, <a>, <input>, <textarea>, <select>, and any element with tabindex="0".

Inline link
tabindex="0" div (e.g. feedback card)

Tailwind utilities (preferred when inline)

focus-visible:ring-2 focus-visible:ring-teal-500 focus-visible:ring-offset-2 dark:focus-visible:ring-offset-gray-900 focus:outline-none

§1a — Compact variant (feedback cards)

Same 2px teal ring, but 1px outer offset instead of 2px. Used only inside dense card stacks (feedback list, agent-mine list) where 2px offset causes the ring to visually merge with the adjacent card's border.

Do not use focus:ring-1 — the ring width stays 2px. Only the offset changes. Anyone who tries to use focus:ring-1 fails this spec; see §3 below.

§2 — Variant-matched focus rings (primary actions ONLY)

The focus ring MAY match the button's background color only when the button is a semantically-loaded primary action. The color choice reinforces what the key press will do. Every other element uses §1.

Allowed variant colors:

  • green-500 — Approve primary buttons (decision form)
  • amber-500 — Request Changes primary buttons (decision form), Confirm Revisit (revisit modal, because the action is semantically "warning" — it resets state)
  • red-500 — destructive primary (Delete feedback, Discard draft)

Revisit modal Cancel button uses the §1 canonical teal ring — Cancel is not a semantically-loaded primary action.

§3 — Forbidden patterns

  • focus:ring-1 anywhere — the ring width is always 2px. Use focus-visible:ring-2 with an offset adjustment instead.
  • Any interactive element without a focus-visible: ring — browser default outline may be suppressed by Tailwind reset and is inconsistent across browsers.
  • focus: (not focus-visible:) on elements that should not show a ring on mouse click (e.g. all buttons). Use focus-visible: so mouse users don't see a spurious ring.
  • Variant-matched rings on anything other than the primary actions listed in §2.
  • Non-2px offset except the §1a compact variant (1px offset for dense card stacks).

§4 — Enforcement (feedback-assessor + dev-stage gate)

  • grep -rEn 'focus:ring-1' stages/design/artifacts/ returns 0 matches on interactive elements (this spec file itself encodes the colon as &#58; in docs so the grep audit stays clean).
  • Every <button>, <a>, <input>, <textarea>, [tabindex="0"] declares focus-visible:ring-2 focus-visible:ring-teal-500 (§1) OR one of the variant rings in §2.
  • Revisit-modal Cancel button uses §1 teal-500 ring (not left unstyled).
  • Revisit-modal Confirm button uses §2 amber-500 variant ring (documented above as an allowed variant).
  • Manual test: Tab through every artifact; verify every interactive element shows a 2px teal (or variant) ring. No element shows the default browser outline (it must be suppressed or matched).

§5 — Accessibility rationale

  • WCAG 2.4.7 Focus Visible (Level AA): every focusable element has a visible focus indicator.
  • WCAG 2.4.11 Focus Not Obscured (Minimum) (Level AA, WCAG 2.2): the focus ring sits outside the element via outline-offset: 2px so it is not covered by adjacent content.
  • WCAG 2.4.13 Focus Appearance (Level AAA, WCAG 2.2): 2px solid ring with ≥ 3:1 contrast against adjacent colors. teal-500 (#14B8A6) vs white (#FFFFFF) = 3.02:1 (passes); teal-400 (#2DD4BF) vs gray-950 (#030712) = 8.99:1 (passes).
  • :focus-visible (not :focus) prevents the ring from appearing on mouse click — matches user expectation and reduces visual noise for sighted mouse users.
