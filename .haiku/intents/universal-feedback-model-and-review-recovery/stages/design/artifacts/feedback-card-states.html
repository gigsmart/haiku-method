4. Origin Badge Visual Inventory

Every feedback card shows a small origin pill at top-left so the reviewer can tell at a glance where an item came from. WCAG AA contrast verified for every pair below (text vs. badge bg > 4.5:1 in light, > 4.5:1 in dark).

origin light dark ownership kind WCAG AA
adversarial-review 🔍Review Agent 🔍Review Agent Agent (adversarial reviewer / feedback assessor) PASS
external-pr 🔗External PR 🔗External PR User (GitHub reviewer via provider translation) PASS
external-mr 🔗External MR 🔗External MR User (GitLab reviewer via provider translation) PASS
user-visual ✎User Visual ✎User Visual User (annotation on doc / mockup) PASS
user-chat 💬User Chat 💬User Chat User (typed feedback from review app) PASS
agent 🤖Agent 🤖Agent Agent (system / orchestrator / feedback assessor) PASS

Rose is reserved for the review-agent specifically (distinct "adversarial" voice). External PR & MR share the violet palette because they fill the same role from the reviewer's perspective — both are "a human left this on the VCS platform". Sky covers user-originated items whether visual or text.

5. Optimistic UI Pattern

Step 1 · Optimistic update

User clicks Verify & Close. Card status badge flips to closed immediately, left-border turns green, footer buttons update to show Re-open. A brief feedback-status-change flash (0.4s) confirms the transition visually.

Step 2 · Tool call in flight

Footer buttons enter disabled state with a spinner in the header row. The card sets aria-busy="true". If the call resolves OK, the disabled state clears and the card stays in its new status.

Step 3 · On failure — revert

If the tool call rejects, the card reverts to its previous status (badge, border, buttons). A toast appears bottom-right with the guard-specific copy below. Card shakes briefly (0.2s horizontal translate) to draw attention.

6. Toast Copy — One Per Guard Outcome

When the tool layer rejects a transition, the review app renders one of these toasts. Copy is explicit about why — no generic "something went wrong".

guard outcome code toast title toast body
not_user_initiable Agents mark that — not reviewers "pending → addressed and pending → closed are owned by the agent. This card will update automatically when the fix lands or the gate verifies."
invalid_transition That move isn't valid from here "You can't go from {from} to {to} directly. Re-open it first, then make your change."
stale_version Someone else moved faster "The card changed to {server_status} while you were looking. Your action didn't apply — refresh and decide again."
author_type_mismatch Only a human can do that "This transition requires a user session. The tool rejected it because the caller's author_type is agent."
feedback_not_found Feedback item missing "Couldn't find {fbId} on disk. It may have been deleted or renamed — reload the page."
network_error Can't reach the MCP server "Your change didn't go through. Check that the H·AI·K·U review server is still running, then try again."

Toast spec: role="status" container, aria-live="polite", auto-dismiss at 6s with a close button for manual dismissal. On failure toasts, auto-dismiss extends to 8s. All toasts render above the card stack at z-50.

7. Responsive & Accessibility Requirements

Mobile (375px)
  • Footer buttons min-height 44px
  • Buttons stack to full-width below card body
  • Origin + status badges wrap on new row when they would overflow
Tablet (768px) & Desktop (1280px)
  • Footer buttons compact 28px height, right-aligned
  • Badges remain on same row as title
  • Card hover border brightens to teal-400 / teal-500
Accessibility
  • Cards are <article> with tabindex="0"; ENTER opens detail
  • Every footer button has aria-label including FB-id and action
  • Status and origin are read via text inside badges (not color-only)
  • In-flight cards set aria-busy="true"; toasts are role="status" aria-live="polite"
  • Focus ring: 2px teal-500 outline with 2px offset; matches existing tab focus pattern
