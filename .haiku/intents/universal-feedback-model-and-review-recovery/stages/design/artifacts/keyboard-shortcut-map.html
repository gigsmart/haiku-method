2. Esc precedence ladder

Exactly one rung is consumed per press. Earlier rungs short-circuit later ones.

  1. 1

    Modal open (e.g. revisit modal)

    Close the modal. Focus returns to the element that opened it.

  2. 2

    Annotation popover open

    Close popover; any unsaved draft is discarded (matches mockup behavior).

  3. 3

    Help overlay open

    Toggles overlay closed (same as pressing ? again).

  4. 4

    Focus inside an input / textarea / contenteditable

    Call target.blur(). Do not also dismiss an outer layer — return early.

  5. —

    No layer to dismiss

    No-op. Do not steal the native browser behavior (e.g. canceling pending forms).

3. Browser, OS & screen-reader conflict analysis

Every single-key shortcut is checked against the Chrome, Firefox, and Safari default bindings and against the virtual-cursor ("browse-mode") single-key navigation shortcuts of NVDA, JAWS, and VoiceOver. Any chord using a modifier (Cmd/Ctrl/Alt/Shift) is explicitly avoided so we never intercept platform bindings.

3a. Browser / OS defaults

Key Browser / OS default Our binding Resolution
/ Chrome / Firefox: "quick find in page". Safari: no default at the top level. Focus feedback textarea Accepted. Industry-standard for "focus search" (GitHub, Slack, Linear). We call preventDefault() so the browser's quick-find does not open.
? No browser default (Shift+/). Some OS screen readers intercept. Toggle help overlay Accepted. Must be suppressed while focus is in an input (otherwise the user typing "?" would open the overlay). Enforced by the inInput guard.
j k No defaults. Vim muscle-memory convention. Focus next / previous feedback Accepted.
[ ] No browser defaults. Some IME layouts map these to accents (deadkey). Prev / next stage Accepted. IME deadkey users can switch stages via the visible stage strip (mouse / Tab).
a r n No browser defaults. Approve / Reopen-or-Request / Next-unseen Accepted.
g No browser default. Gmail-style two-key leader. Latch; awaits o/u/k/p A bare g with no completion within 1 s is silently discarded (no visual arming indicator — low-signal and non-intrusive). If the user hits g twice, the second press resets the 1 s timer.
Enter Native: activate focused button / submit form. Cross-flash focused feedback Accepted only when a feedback card is focused and no input is active. Native button / form submit is not intercepted — the focused-feedback path short-circuits before native activation because document.activeElement is the feedback card (not a button).
Esc Native: cancel dialog / exit fullscreen / blur. Layered dismiss (see §2) We do not call preventDefault() on the no-op rung so fullscreen and native dialogs still exit cleanly when no app layer is open.

3b. Screen-reader browse-mode conflicts

Screen readers expose a virtual cursor ("browse mode" in NVDA/JAWS, "QuickNav" in VoiceOver) that intercepts bare single-key presses for structural navigation (jump to next heading, link, region, etc.) before the page's keydown handler ever sees them. The page cannot detect browse mode from JavaScript — the input-capture rule (§4) does not cover it. This table names the specific conflicts so development knows what to test.

App shortcut NVDA browse-mode JAWS virtual-cursor VoiceOver QuickNav Mitigation
j / k No default No default No default Safe.
r Next region (landmark) Next region (landmark) No default Conflict. In browse mode the app never sees r — the user moves between landmarks instead. In focus mode the app captures it. Users must switch modes manually (NVDA: Insert+Space). Documented in help overlay; also offered as a remappable shortcut via the modifier-key setting below.
n Next non-linked text Next non-linked text No default Minor conflict in browse mode; app semantics ("next unseen artifact") are a visual-hierarchy concept so browse-mode intercept is acceptable. Modifier-key setting remaps to Alt+n.
a / c No default No default No default Safe. All three readers reserve capital-letter variants (e.g. Shift+a for "previous annotation" in JAWS); we only bind the lowercase form.
/ Next form field Next form field No default Conflict. The screen-reader shortcut actually does what the user wanted (jumps to the sidebar textarea is roughly equivalent to "next form field"), so the conflict is benign. Documented.
? No default No default Rarely intercepted Mostly safe. VoiceOver users with custom commander bindings can collide; documented as known-caveat in help overlay.
[ / ] No default No default No default Safe.

3c. User setting: "Require modifier key for shortcuts" (default: off)

  • Setting exposed in the help overlay footer and in the user's account preferences. When toggled on, every bare single-key shortcut is remapped to require Alt as a prefix: j → Alt+j, r → Alt+r, a → Alt+a, c → Alt+c, etc. The g+key two-key sequences become Alt+g+key.
  • Why Alt? Cmd/Ctrl collides with browser defaults (Ctrl+a = select all, Cmd+r = reload). Alt is not claimed by the SR virtual cursor and is almost never claimed by browser defaults on the target platforms.
  • Esc / Enter / ? (Shift+/) / [ / ] are not remapped — they already require or implicitly involve a modifier, or they are unique punctuation.
  • Setting persists in localStorage under review-ui.shortcutsRequireModifier. On enabling, the help overlay's shortcut chips show the Alt prefix inline.
  • Auto-detection is not used. The WebAIM-recommended aria-live screen-reader probe is unreliable (false positives from live regions in apps without a reader). Users who rely on an SR opt in explicitly; users who don't aren't penalized.

3d. aria-keyshortcuts contract

  • Every element that responds to a shortcut (sidebar feedback card, Approve button, Request-changes button, tab button, stage strip button, general-comment textarea, pin marker) carries an aria-keyshortcuts="…" attribute listing the chord in W3C format (space-separated modifiers + main key, e.g. "Enter", "Alt+j", "c").
  • When "Require modifier key" is on, the attribute value is updated to the Alt-prefixed chord so the reader announces the correct binding (e.g. "Alt+c").
  • The help overlay itself uses role="dialog" aria-modal="true" aria-keyshortcuts="?" on the trigger.
  • Pin markers declare aria-keyshortcuts="Enter" on focus so readers announce "Feedback FB-12 at 42% 60% — press Enter to view details".

4. Input-capture rule (explicit)

Guard: inInput = tag === 'INPUT' || tag === 'TEXTAREA' || target.isContentEditable

  • When inInput === true, all shortcuts are suppressed except Esc.
  • Esc while in an input blurs the input (does not dismiss outer modal / popover). Precedence rung 4 short-circuits rungs 1–3 when focus is in an input.
  • This means the user can type ?, /, a, r, j, k, etc. inside the feedback textarea without triggering shortcuts.
  • Platform shortcuts that the browser delivers with a modifier (Cmd+Enter to submit a form, for example) are not intercepted; we check e.metaKey || e.ctrlKey || e.altKey and return early if any modifier is set (except where a chord specifically requires one — none currently do).

6. Tab focus order

Tab / Shift+Tab walk the DOM in reading order. No manual tabindex values above 0. Every visible interactive element is reachable. See the full specification in focus-order-spec.md.

  1. Skip links (first focusable elements, visually hidden until focused) — Skip to main content, Skip to feedback list. Anchors: #main-content, #feedback-list. See skip-link-spec.html.
  2. Theme toggle (header)
  3. Stage strip buttons — inception, design, product, development (upcoming stages are disabled and skipped by native Tab)
  4. Tabs: Overview, Units, Knowledge, Outputs
  5. Main content interactive elements in reading order including pin markers interleaved in reading order (FB-17 fix: pins are tabindex="0" <button> elements with aria-label="Feedback FB-XX at X% Y% — press Enter to view details" and aria-keyshortcuts="Enter"). Also: expand toggles, "Next unseen" buttons, artifact links.
  6. Sidebar: feedback filter pills → feedback cards (tabIndex={0}) → expanded-card action buttons
  7. Sidebar general-comment textarea
  8. Sidebar decision buttons: Approve, External Review, Request Changes
  9. Help-overlay button (if rendered)

Pin ↔ sidebar card reciprocity (FB-17)

  • Enter on a focused pin → expands the linked sidebar feedback card and scrolls it into view; applies .fb-flash to the sidebar card.
  • Enter on a focused sidebar feedback card → cross-flashes the corresponding pin (applies .unit-flash + scroll-into-view on the pin).
  • Both directions are documented here and implemented through the same shared crossFlash(fbId) helper (see §7).

7. Implementation contract for development stage

Developers wire the behavior defined above using these exact hooks. The function names mirror the mockup so the diff is small.

  • document.addEventListener('keydown', handler) on mount, cleanup on unmount.
  • Module-level state: gPending: boolean, gPendingTimer: number | null, focusedFbId: string | null.
  • Per-feedback-card: tabIndex={0}, data-fb-id, role="button", aria-pressed reflecting focus.
  • Focus ring: CSS classes .fb-card + .fb-card.is-focused; no inline styles. See focus-ring-spec.html.
  • Guard ordering in keydown: (1) Escape — layered dismiss; (2) early-return if inInput; (3) resolve gPending; (4) switch on e.key.
  • All app shortcuts call e.preventDefault() on match so browser defaults do not fire.
  • Reference behavior lives verbatim in review-ui-mockup.html lines 1826–1910.

8. Accessibility notes

  • Every shortcut has a visible UI alternative (stage buttons, tab buttons, approve / request buttons, sidebar cards).
  • Help overlay uses role="dialog" aria-modal="true"; focus traps within; on close, focus returns to the element that opened it.
  • Focus indicator is never removed — we extend the native focus with a teal ring. Users with prefers-reduced-motion: reduce skip the cross-flash animation; the ring remains visible.
  • Screen readers receive a live-region announcement when focus changes feedback ("Feedback FB-03, pending, adversarial review, focused"). Announcement goes through a role="status" aria-live="polite" region on the sidebar.
  • The help-overlay footer reminds keyboard users that shortcuts are suppressed inside inputs.
