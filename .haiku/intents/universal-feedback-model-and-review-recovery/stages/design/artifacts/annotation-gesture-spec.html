Design Stage · unit-04

Annotation gesture spec

closes FB-01

1. What this spec covers

Today the review UI renders existing annotations but does not specify how a reviewer creates one. This unit closes that gap. Every artifact kind in the feedback surface — raster image, inline SVG, markdown / text, HTML iframe, PDF embed — gets a first-class creation gesture, a cursor affordance, and a popover contract.

The deliverable is concrete enough for development to wire up without inventing UX. See annotation-popover-states.html for the visual states of the popover itself.

2. Gesture matrix — one row per artifact kind

Artifact kind MIME / detector Cursor affordance Creation gesture Captured location
Raster image image/* cursor-crosshair Click anywhere on the image wrapper { x, y } ∈ [0,1]
Inline SVG image/svg+xml (inlined) cursor-crosshair Click anywhere on the SVG canvas wrapper { x, y } ∈ [0,1]
Markdown / text text/markdown · text/plain cursor-pointer on hover row Click a line (or focus + A) { line } (1-indexed)
HTML iframe text/html (sandboxed) iframe is non-interactive for annotation; adjacent form is keyboard reachable Use "Add feedback on this preview" button beside the iframe → fills location form { region } (user string)
PDF embed application/pdf iframe is non-interactive; adjacent form is keyboard reachable Use "Add feedback on this preview" button → fills location form { page, region }

Any future artifact kind inherits the matching pattern: if it's spatial, it gets cursor-crosshair + {x, y}; if it's line-indexable text, it gets {line}; if it's embedded in a sandboxed iframe, it gets the location-form fallback.

3. Spatial annotation — raster image

Click-to-place. Move the mouse to see a ghost pin follow the cursor. Click to open the popover anchored at that location. Normalized {x, y} ∈ [0,1] so coordinates survive resize/rescale.

Touch behavior

Long-press (≥400ms) replaces click-to-place on touch devices — avoids the scroll-collision. Tapping an existing pin opens that feedback.

Rejection bounds

If x or y falls outside [0,1] (edge click miss), the gesture is silently discarded — no popover, no error toast.

4. Spatial annotation — inline SVG

Identical gesture to raster images — wrapper div receives the click, image contents are pointer-events: none so the whole surface is a uniform click target. No special SVG-element click handling in v1.

<div class="relative cursor-crosshair" onclick="startAnnotationSpatial(...)">
  <svg ... class="pointer-events-none" />
  {/* overlay pins positioned absolutely */}
</div>

5. Line-based annotation — markdown / text

Each line is a row in a 2-column grid (gutter + content). Hover highlights the entire row in teal; click anywhere on the row opens the popover anchored near the click. Keyboard path: arrow-keys focus a line, A opens popover at that line.

6. Iframe fallback — HTML / PDF

Same-origin iframes can in principle receive click handlers, but sandboxed srcdoc and cross-origin PDF viewers cannot. We treat all iframe-embedded artifacts uniformly: direct click-in is out. Reviewers submit a location through an adjacent form attached below the preview.

Add feedback on this preview

The popover opens with the captured {page, region} preselected so the reviewer only types title + body.

Why not click-through?

  • Sandboxed srcdoc: sandbox="allow-same-origin" blocks parent script access; we can't attach a click listener inside.
  • Cross-origin PDFs: the PDF viewer consumes all mouse events; the parent never sees them.
  • Consistency: one fallback pattern across both kinds beats two custom integrations. When pdf.js lands in v2 we can add {page}-coordinate click-through without breaking the form.

8. Accessibility contract

  • Focus trap: once the popover opens, focus is trapped between title → body → Cancel → Create → (loop).
  • Focus return: closing the popover (save, cancel, ESC, click-outside) returns focus to the element that triggered creation — the artifact wrapper, the line row, or the iframe form's submit button.
  • ARIA: popover is role="dialog" aria-modal="true" aria-labelledby="ann-popover-label"; title input has aria-required="true".
  • Live-region announcement: on save, a role="status" aria-live="polite" region announces "Feedback FB-NN created at <location label>".
  • Touch targets: every button in the popover footer is ≥ 44×44px on mobile (see annotation-popover-states.html → small-viewport state).
  • Cursor affordance: cursor-crosshair on spatial wrappers and cursor-pointer on line rows is a visual hint — screen readers get role="button" or equivalent regardless.
  • No color-only signals: the ghost pin uses a dashed teal border + always-on fill; pin status (pending / addressed / closed) is encoded as both color + number label.

9. Storage contract

All created annotations persist through haiku_feedback with the exact shape below. The CRUD tool MUST accept this payload; the reviewer MUST produce it from the gesture data. 

// Feedback payload — annotation creation
{
  title: "Review gate timing unclear",
  body:  "Gate rolls back to elaborate but…",
  origin: "user-visual",     // always user-visual for UI-created
  target: {
    kind: "output",           // "unit" | "knowledge" | "output" | "file"
    outputName: "review-ui-mockup.html",
    annotation: {
      // EXACTLY ONE of the three shapes below

      // (a) spatial — image, svg
      x: 0.42, y: 0.60     // both ∈ [0, 1]

      // (b) line — markdown, text
      // line: 3                 // 1-indexed

      // (c) region — html, pdf
      // page: 3,                // optional, PDF only
      // region: "top-right CTA" // required when no page/line/x
    }
  }
}

Invariants the server MUST enforce

  • target.annotation is required for origin: "user-visual". Reject the request without it.
  • Exactly one of {x,y}, {line}, {page,region}, or {region} must be present. Mixing shapes is rejected.
  • x and y MUST be clamped to [0,1] on the server side too — never trust the client.
  • line MUST be a positive integer ≥ 1.
  • region MUST be a non-empty string when present (trim-then-check).

10. Creation flow — end-to-end

① Gesture

User clicks image/svg/line or submits iframe form.

→

② Capture

Compute {x,y} | {line} | {page,region}. Stash in annContext.

→

③ Popover

Anchor popover near cursor (or form); focus title field.

→

④ Save

POST to haiku_feedback with target + annotation; append to sidebar.

No intermediate "draft" state in v1 — save is a single transactional write. If the request fails, the popover stays open with a red toast inline ("Couldn't save — retry?") and the captured location is preserved so the reviewer doesn't lose their placement.
